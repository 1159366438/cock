# Attendance System 安全与质量审计报告与修复任务清单

## 执行摘要

本报告依据大厂安全与质量规范，对 Attendance System 进行了全面的代码审计与架构评估。本次审计共识别 **15 类** 问题，涵盖安全、并发、架构、测试、部署五大领域。根据风险等级及业务影响，将问题划分为 P0（立即修复）、P1（短期修复）、P2（中期修复）、P3（长期规划）四个优先级。本报告同时包含详细的可执行修复任务清单，供开发团队逐项落实。

---

## 1. 安全性问题（P0 - 立即修复）

### 1.1 认证与授权机制

| 问题 | 风险等级 | 影响说明 |
|------|----------|----------|
| JWT Token 机制被注释，当前依赖 Session 认证 | **严重** | 会话状态依赖服务端存储，无法水平扩展；易受 CSRF 攻击 |
| 缺少细粒度权限控制 | **严重** | 用户可能访问未授权资源，存在越权风险 |

**修复任务**  

**任务1：启用 JWT Token 认证机制**  
- **方案**：添加 JWT 依赖，配置 `SecurityConfig` 禁用 Session，实现 `JwtTokenUtil` 生成/解析 Token，添加 `JwtAuthenticationFilter` 从请求头提取并校验 Token。  
- **涉及模块**：`backend/src/main/java/com/example/config/SecurityConfig.java`，新增 `JwtTokenUtil`、`JwtAuthenticationFilter`。  
- **预估工时**：1 人天  
- **验收标准**：
  - 登录接口返回 JWT Token。
  - 携带有效 Token 可正常访问受保护接口。
  - 携带无效/过期 Token 返回 401。
  - 不使用 Session（`JSESSIONID` 不再生成）。

**任务2：实现细粒度权限控制（RBAC）**  
- **方案**：设计权限表结构（`role`、`permission`、`user_role`、`role_permission`），启用 `@EnableGlobalMethodSecurity(prePostEnabled = true)`，在 Controller 方法上使用 `@PreAuthorize` 注解。编写权限加载组件，从数据库获取用户权限。  
- **涉及模块**：`backend/src/main/java/com/example/entity/`、`service/`、`controller/`。  
- **预估工时**：2 人天  
- **验收标准**：
  - 不同角色的用户登录后，只能访问授权接口，无权限接口返回 403。
  - 权限数据可通过数据库配置。

### 1.2 输入校验缺失

| 问题 | 风险等级 | 影响说明 |
|------|----------|----------|
| 参数校验不严格，缺少长度、格式、类型限制 | **严重** | 可能导致 SQL 注入、XSS、缓冲区溢出等攻击 |
| 未对用户输入进行白名单校验 | **高危** | 恶意输入可能绕过业务逻辑 |

**修复任务**  

**任务3：严格参数校验**  
- **方案**：引入 `spring-boot-starter-validation`，在 DTO 类中使用 `@NotNull`、`@NotBlank`、`@Size`、`@Pattern` 等注解；Controller 方法参数前添加 `@Valid`；全局异常处理器增加 `MethodArgumentNotValidException` 处理；对字符串参数进行 HTML 转义（可使用 `HtmlUtils.htmlEscape`）。  
- **涉及模块**：所有 Controller、DTO、全局异常处理器。  
- **预估工时**：1 人天  
- **验收标准**：
  - 提交非法参数（空值、超长、特殊字符）时返回 400 错误，并提示具体字段错误。
  - 输入包含 `<script>` 等特殊字符时，存储后在前端展示为文本（防止 XSS）。

### 1.3 敏感信息泄露

| 问题 | 风险等级 | 影响说明 |
|------|----------|----------|
| 异常堆栈直接返回给客户端 | **中危** | 暴露系统路径、技术栈信息，为攻击者提供线索 |
| 密码重置功能缺失 | **中危** | 用户体验差，存在账号锁定风险 |

**修复任务**  

**任务4：完善全局异常处理**  
- **方案**：在 `GlobalExceptionHandler` 中添加 `@ExceptionHandler(Exception.class)`，记录日志并返回通用错误码，不暴露内部堆栈。  
- **涉及模块**：`backend/src/main/java/com/example/config/GlobalExceptionHandler.java`。  
- **预估工时**：0.5 人天  
- **验收标准**：
  - 任何未捕获异常，返回 `{"code":500, "msg":"系统繁忙，请稍后再试"}`，控制台记录详细堆栈。
  - 不暴露任何内部信息（如 SQL 语句、文件路径）。

**任务5：实现密码重置功能**  
- **方案**：增加密码重置接口（通过邮箱/手机验证码），生成临时 Token 用于重置密码。  
- **涉及模块**：`UserController`、`UserService`，新增验证码服务。  
- **预估工时**：2 人天  
- **验收标准**：
  - 用户可通过邮箱/手机号申请重置密码。
  - 重置链接或验证码有效期内可修改密码。

## 2. 并发与性能问题（P0 - 立即修复）

### 2.1 打卡并发控制

| 问题 | 风险等级 | 影响说明 |
|------|----------|----------|
| 缺少分布式锁，用户可在极短时间内多次打卡 | **高危** | 造成数据不一致，影响考勤统计准确性 |
| 未对同一时段打卡做防重处理 | **高危** | 存在恶意刷打卡记录的风险 |

**修复任务**  

**任务6：打卡并发控制（防重复打卡）**  
- **方案**：引入 Redis，在打卡接口中先检查 `user:userId:today` 键是否存在，若存在则返回“今日已打卡”；使用 Redis 分布式锁 `SET user:lock:userId 1 EX 1 NX` 保证同一时刻只有一个请求处理打卡；数据库增加唯一约束 `UNIQUE (user_id, DATE(check_in_time), check_in_type)` 作为双重保障。  
- **涉及模块**：`PunchController`，新增 Redis 配置类。  
- **预估工时**：1 人天  
- **验收标准**：
  - 同一用户一天内多次调用打卡接口，只有第一次成功，后续返回错误信息。
  - 高并发场景下（压力测试）不会产生重复打卡记录。

### 2.2 数据库性能瓶颈

| 问题 | 风险等级 | 影响说明 |
|------|----------|----------|
| 缺少必要索引，存在全表扫描 | **高危** | 查询响应慢，数据库 CPU 负载高 |
| N+1 查询问题 | **中危** | 页面加载慢，影响用户体验 |

**修复任务**  

**任务7：数据库索引优化**  
- **方案**：编写迁移脚本，为 `punch_record` 表增加 `(user_id, check_in_time)` 联合索引，为 `user` 表增加 `department_id` 索引。  
- **涉及模块**：数据库迁移脚本。  
- **预估工时**：0.5 人天  
- **验收标准**：
  - 高频查询 `EXPLAIN` 显示使用索引。
  - 大数据量下（模拟 10 万条记录），查询响应时间 < 200ms。

**任务8：SQL 优化（N+1 查询问题）**  
- **方案**：检查 MyBatis Mapper XML 中的查询，使用 `JOIN` 一次获取关联数据，避免循环查询。  
- **涉及模块**：`backend/src/main/resources/mapper/PunchRecordMapper.xml`。  
- **预估工时**：0.5 人天  
- **验收标准**：
  - 开启 SQL 日志，查看查询次数；列表查询只执行 1-2 条 SQL，而非 1+N 条。

## 3. 代码质量与架构（P1 - 短期修复）

### 3.1 异常处理机制

| 问题 | 风险等级 | 影响说明 |
|------|----------|----------|
| 全局异常处理器未覆盖所有异常类型 | **中危** | 部分异常仍会暴露给前端，影响用户体验 |
| 缺少熔断降级机制 | **中危** | 下游依赖故障时可能引起雪崩 |

**修复任务**  

**任务9：完善全局异常处理**（已在 P0 任务4中覆盖，此处不再重复）  

**任务10：引入熔断降级机制**  
- **方案**：添加 `resilience4j-spring-boot2` 依赖，在调用 AI 服务等外部依赖的方法上添加 `@CircuitBreaker` 注解，配置失败率阈值，定义 fallback 方法。  
- **涉及模块**：调用外部服务的 Service 类。  
- **预估工时**：1 人天  
- **验收标准**：
  - 模拟外部服务超时或返回错误，达到阈值后触发熔断，后续请求直接走 fallback。
  - 熔断恢复后，能正常调用。

### 3.2 缓存策略

| 问题 | 风险等级 | 影响说明 |
|------|----------|----------|
| 未引入 Redis 缓存 | **中危** | 热点数据（部门树、用户权限）频繁查库 |
| 会话存储方式不当 | **中危** | 无法支持多实例部署 |

**修复任务**  

**任务11：引入 Redis 缓存**  
- **方案**：配置 Redis 连接；使用 `@Cacheable` 缓存部门树列表和用户权限信息；使用 `@CacheEvict` 在数据变更时清除缓存。  
- **涉及模块**：`DepartmentService`、`UserService`。  
- **预估工时**：1 人天  
- **验收标准**：
  - 第一次请求部门树后，第二次相同请求直接走缓存，不查询数据库。
  - 更新部门后，缓存自动清除，下次查询重新加载。

**任务12：统一会话管理**  
- **方案**：如果已启用 JWT 无状态认证，则无需额外 Session 管理。若仍需 Session，可配置 `spring.session.store-type=redis` 实现 Redis 存储 Session。  
- **涉及模块**：`application.yml`。  
- **预估工时**：0.5 人天  
- **验收标准**：
  - 多实例部署时，登录状态可共享。

## 4. 配置与部署（P2 - 中期修复）

### 4.1 环境配置管理

| 问题 | 风险等级 | 影响说明 |
|------|----------|----------|
| 开发与生产配置混用 | **低危** | 存在误操作风险，如连接错误数据库 |
| 敏感信息硬编码 | **中危** | 数据库密码、密钥等信息可能泄露 |

**修复任务**  

**任务13：环境配置分离**  
- **方案**：创建 `application-dev.yml`、`application-prod.yml`，将不同环境的配置分别放入；启动时通过 `spring.profiles.active` 指定环境。  
- **涉及模块**：`backend/src/main/resources/`。  
- **预估工时**：0.5 人天  
- **验收标准**：
  - 不同 profile 启动时，连接对应的数据库。
  - 配置文件无明文密码。

**任务14：敏感信息脱敏**  
- **方案**：将数据库密码、JWT 密钥等敏感信息改为从环境变量读取；在日志中避免打印密码字段；使用 `@JsonIgnore` 确保密码不序列化到 JSON。  
- **涉及模块**：`User` 实体类，日志打印处，`application.yml`。  
- **预估工时**：0.5 人天  
- **验收标准**：
  - 登录、查询用户等接口响应不包含 `password` 字段。
  - 日志中无明文密码。

### 4.2 监控与告警

| 问题 | 风险等级 | 影响说明 |
|------|----------|----------|
| 缺少 APM 监控 | **中危** | 无法实时发现性能瓶颈 |
| 无健康检查接口 | **低危** | 无法感知服务状态 |

**修复任务**  

**任务15：集成 APM 监控**  
- **方案**：下载 SkyWalking Java Agent，在启动参数中添加 `-javaagent:/path/to/skywalking-agent.jar`，配置服务名称和后端地址。  
- **涉及模块**：部署脚本、启动参数。  
- **预估工时**：1 人天  
- **验收标准**：
  - SkyWalking UI 能展示服务拓扑、接口列表及调用链。

**任务16：暴露健康检查接口**  
- **方案**：引入 `spring-boot-starter-actuator`，配置 `management.endpoints.web.exposure.include=health,info`。  
- **涉及模块**：`pom.xml`，`application.yml`。  
- **预估工时**：0.5 人天  
- **验收标准**：
  - 访问 `/actuator/health` 返回 `{"status":"UP"}`。

## 6. 前端安全（P1 - 短期修复）

### 6.1 XSS 防护缺失

| 问题 | 风险等级 | 影响说明 |
|------|----------|----------|
| 未对用户输入进行转义 | **高危** | 恶意脚本可能被注入并执行 |
| DOM 操作存在安全风险 | **中危** | 动态插入内容可能触发 XSS |

**修复任务**  

**任务19：XSS 防护**  
- **方案**：前端安装 `dompurify` 库，在展示用户输入内容时使用 `DOMPurify.sanitize(html)` 过滤；避免使用 `v-html` 渲染不可信内容；后端在返回 JSON 前也可对字符串字段进行 HTML 转义。  
- **涉及模块**：前端所有展示用户输入的地方。  
- **预估工时**：1 人天  
- **验收标准**：
  - 输入 `<script>alert('xss')</script>`，保存后再次展示时，页面不弹出弹窗，仅显示纯文本。

### 6.2 客户端验证不足

| 问题 | 风险等级 | 影响说明 |
|------|----------|----------|
| 前端验证逻辑可被绕过 | **中危** | 恶意用户可提交不符合要求的数据 |

**修复任务**  

**任务20：前后端验证统一**  
- **方案**：确保后端验证规则与前端保持一致；前端表单提交时进行初步校验提升体验，后端作为最终防线。  
- **涉及模块**：前端表单组件、后端 Controller 参数校验。  
- **预估工时**：1 人天  
- **验收标准**：
  - 通过 Postman 绕过前端发送非法参数，后端返回 400 错误。
  - 前端在输入时能及时提示错误。
